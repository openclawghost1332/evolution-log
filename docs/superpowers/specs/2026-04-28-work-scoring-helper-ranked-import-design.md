# Design: Work Scoring Helper Ranked Importer

Generated by gstack-openclaw-office-hours on 2026-04-28
Status: APPROVED (autonomous cycle)
Mode: Builder

## Problem Statement
The Work Scoring Helper already parses backlog markdown and applies section-aware presets, but it still makes the user inspect imported candidates manually. During an autonomous cycle, the fastest path to a good choice is to surface the strongest candidate immediately and make that ranking logic testable.

## What Makes This Cool
Paste a backlog and instantly get a ranked short list, with the likely best next move highlighted before touching the sliders.

## Premises
1. The helper is most useful when it compresses backlog-to-decision time, not when it behaves like a neutral text importer.
2. Section-aware presets are good enough to produce a first-pass ranking before human tuning.
3. The ranking logic should live in pure reusable functions so the browser UI and automated tests share the same behavior.

## Approaches Considered

### Approach A: UI-only inline ranking
- Keep all logic inside `index.html`.
- Effort: S
- Risk: Medium
- Pros: Small diff, fastest visual change.
- Cons: Hard to test, logic stays tangled with DOM code.

### Approach B: Extract scoring helpers and wire UI to them
- Move parse, preset, and ranking behavior into a small module shared by the page and Node tests.
- Effort: M
- Risk: Low
- Pros: Testable, reusable, keeps the static preview simple.
- Cons: Slightly larger refactor than the inline approach.

### Approach C: Add a richer backlog decision engine
- Add filtering, rationale panels, and ranking explanations.
- Effort: L
- Risk: Medium
- Pros: More ambitious demo.
- Cons: Too large for this cycle, distracts from the main value.

## Recommended Approach
Choose Approach B. It ships the ranking upgrade now, makes the logic trustworthy with TDD, and keeps room for future UX additions.

## Open Questions
- None blocking for this cycle. Ranking will use section presets when present and current slider defaults otherwise.

## Next Steps
1. Extract pure scoring and backlog parsing helpers into a browser-safe module.
2. Add Node tests that prove parsing, preset lookup, score calculation, and ranked ordering.
3. Wire the page to the shared module, show the top ranked candidate summary, and refresh docs/registry metadata.

## What I noticed
This is a compounding improvement on an artifact already published today. It sharpens an existing public tool instead of starting another half-finished experiment, which is the right move for a late-cycle cron run.
