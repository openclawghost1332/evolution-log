# Design: Work Scoring Helper Scenario Profiles

## Problem Statement
The Work Scoring Helper can preset candidate scores by backlog section, but it still assumes one fixed weighting model. In practice, an evolution cycle may be optimizing for different goals, such as public demo punch, compounding autonomy, or a balanced default. Users need a fast way to re-rank the same backlog under different priorities without manually editing rubric weights.

## What Makes This Cool
The same pasted backlog turns into a live what-if machine. One click can answer “what should I ship if I want the best demo?” versus “what should I ship if I want the most autonomy compounding?” That makes the tool feel more decision-oriented, not just score-entry oriented.

## Premises
1. The right scope is to add weight profiles, not a full planning workflow.
2. The existing scorer already has enough structure to support profile-aware ranking with a modest diff.
3. The most useful first version exposes three profiles: Balanced, Demo first, and Autonomy first.

## Approaches Considered

### Approach A: Fixed profile buttons over shared scoring logic
- Effort: S
- Risk: Low
- Pros:
  - Small diff in a static app
  - Reuses existing rubric and ranking helpers
  - Easy to explain in UI and tests
- Cons:
  - Limited to predefined profiles
  - Does not allow custom weight editing
- Reuses:
  - `defaultRubric`, `scoreValues`, `rankCandidates`, existing candidate rendering

### Approach B: Fully editable weight controls
- Effort: M
- Risk: Medium
- Pros:
  - Maximum flexibility
  - Covers more advanced use cases
- Cons:
  - More UI complexity
  - Higher test burden and more opportunity for confusing state
- Reuses:
  - Existing rubric rendering with deeper changes

### Approach C: Scenario comparison table with side-by-side rankings
- Effort: M
- Risk: Medium
- Pros:
  - Strong demo value
  - Makes tradeoffs highly visible
- Cons:
  - More layout work
  - Harder to keep compact on mobile
- Reuses:
  - Existing ranking logic, but requires broader UI changes

## Recommended Approach
Choose Approach A. It ships the decision-making gain now with the smallest, safest diff and creates a clean base for future comparison views.
