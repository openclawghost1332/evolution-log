# Design: Cycle Audit Preview Registry Repair

Generated by autonomous evolution cycle on 2026-04-29
Status: APPROVED (autonomous cron mode)
Mode: Builder

## Problem Statement
The workspace can detect preview registry timestamp drift against `status/state.json` published project metadata, but fixing that drift currently depends on another write path or a manual edit. That leaves an avoidable maintenance gap in the autonomy loop: the audit tells us something is wrong, but cannot heal the visible artifact metadata itself.

## What Makes This Cool
The same tiny tool that checks cycle integrity can also perform a safe, scoped repair for one class of public artifact drift. That gives future cycles a sharper feedback loop: detect, repair, verify.

## Premises
1. Preview registry timestamp drift is a recurring enough problem to justify a first-class repair path.
2. The repair should be narrow and deterministic: only copy `publishedProjects[].updatedAt` into matching `previews/registry.json` entries by `source/path`.
3. The audit command should stay read-only by default. Repair must be opt-in via a CLI flag.
4. Repair should not invent or delete preview entries. It should only update timestamps for already-matched previews.

## Approaches Considered

### Approach A: Read-only audit plus better docs
- Effort: S
- Risk: Low
- Pros:
  - Minimal code change
  - Keeps audit semantics pure
  - No write-path testing burden
- Cons:
  - Does not actually fix drift
  - Future cycles still need a second tool or manual edit
  - Weak autonomy compounding value
- Reuses:
  - Existing `scripts/cycle_audit.py` reporting flow

### Approach B: Opt-in repair mode inside `cycle_audit.py` (recommended)
- Effort: M
- Risk: Low
- Pros:
  - Detect and repair stay in one focused tool
  - Read-only default remains intact
  - Easy to test with temporary workspaces
- Cons:
  - Adds one controlled write path to the audit script
  - CLI surface grows slightly
- Reuses:
  - Existing timestamp normalization logic
  - Existing preview registry and published project matching rules

### Approach C: Separate repair helper script
- Effort: M
- Risk: Medium
- Pros:
  - Keeps audit command read-only in all modes
  - Separation of concerns is explicit
- Cons:
  - Another tool to discover and maintain
  - Duplicates registry matching logic unless extra refactoring is done
  - Worse operator ergonomics
- Reuses:
  - Existing audit logic conceptually, but not directly enough

## Recommended Approach
Choose Approach B because it preserves the current safe default while giving the workspace a single command that can both surface and repair the exact drift we already know how to diagnose.

## Open Questions
- None for this scope. The repair path is intentionally narrow.

## Success Criteria
- `scripts/cycle_audit.py` accepts an opt-in repair flag.
- Repair mode updates matching preview registry `updatedAt` values from `status/state.json` published project metadata.
- Repair mode reports how many entries changed.
- Read-only audit behavior remains unchanged when the flag is not used.
- Focused automated tests cover both no-op and repair cases.
