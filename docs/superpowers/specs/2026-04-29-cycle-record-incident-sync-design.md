# Design: Cycle record incident sync

Generated by brainstorming on 2026-04-29
Status: APPROVED (autonomous evolution cycle)
Mode: Builder

## Problem Statement
`status/state.json` already carries an `incidents` field, but `scripts/cycle_record.py` cannot update it. That leaves incident reporting outside the main cycle-recording path, which creates drift between recorded cycle artifacts and dashboard state.

## What Makes This Cool
One payload becomes the single source of truth for cycle notes, machine records, blockers, published-project sync, preview sync, and incident sync. Future cycles get a cleaner, more automatable finish path.

## Premises
1. Cycle completion should be able to update all user-facing dashboard state that belongs to that cycle.
2. Incident updates should be opt-in, so existing callers that omit incidents keep current behavior.
3. The change should stay minimal and preserve current payload compatibility.

## Approaches Considered

### Approach A: Add optional top-level `incidents` support to cycle records
- Effort: S
- Risk: Low
- Pros:
  - Smallest diff
  - Keeps payload-driven workflow simple
  - Easy to test and document
- Cons:
  - Another top-level optional field in the payload schema
  - No append/merge semantics, only replace-or-preserve
- Reuses: Existing `write_cycle_record` validation and completed-state update flow

### Approach B: Add nested `metadata.state.incidents`
- Effort: M
- Risk: Medium
- Pros:
  - Keeps top-level record schema smaller
  - Leaves room for more future state-sync knobs
- Cons:
  - Harder to discover and document
  - Less consistent with existing top-level `blockers`
- Reuses: Existing metadata plumbing

### Approach C: Separate state patch helper for incidents
- Effort: M
- Risk: Medium
- Pros:
  - Strong separation between artifact writing and dashboard mutation
- Cons:
  - More moving pieces
  - Misses the goal of one authoritative completion payload
- Reuses: Very little

## Recommendation
Choose Approach A because it is the fastest, clearest path to making cycle completion authoritative for incident state without introducing another helper or nested control format.

## Recommended Behavior
- Accept an optional top-level `incidents` list in cycle record payloads.
- Preserve backward compatibility when the field is absent.
- When `--state-mode completed` is used:
  - If `incidents` is present in the payload, replace `status/state.json.incidents` with that list.
  - If `incidents` is absent, preserve the existing `status/state.json.incidents` value.
- Include `incidents` in the emitted JSON record.
- Render an `## Incidents` section in markdown only when the field is present in the payload.

## Testing
- Completed mode updates `incidents` when provided.
- Completed mode preserves existing incidents when omitted.
- JSON record includes `incidents` when provided.
- Markdown includes an incidents section only when provided.
