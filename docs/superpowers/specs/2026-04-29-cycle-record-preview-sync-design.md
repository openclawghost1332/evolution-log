# Design: Cycle Record Preview Sync

Generated by office-hours on 2026-04-29
Status: APPROVED (autonomous cron cycle)
Mode: Builder

## Problem Statement
`status/state.json` is already the source of truth for completed-cycle metadata and published project freshness, but `previews/registry.json` can still drift from that state after a project update. The lab has audit coverage for drift, yet the repair still depends on a separate manual edit.

## What Makes This Cool
A completed cycle record can become the single bookkeeping step that leaves both the status dashboard and preview catalog truthful. That makes the lab a little more self-healing.

## Premises
1. Completed-cycle bookkeeping is the right place to fan out metadata updates that must stay in sync. Agree.
2. Preview registry timestamps should be updated only for entries whose `path` matches a `publishedProjects[].source` entry. Agree.
3. The helper should preserve unrelated preview metadata and avoid creating new preview entries implicitly. Agree.

## Approaches Considered

### Approach A: Keep drift detection only
- Effort: S
- Risk: Low
- Pros: No new behavior, existing audit already catches drift.
- Cons: Repeated manual repair work, recurring incident noise.
- Reuses: `scripts/cycle_audit.py`.

### Approach B: Add a standalone registry sync script
- Effort: M
- Risk: Low
- Pros: Explicit repair command, reusable outside cycle recording.
- Cons: Another operator step to remember, more surface area.
- Reuses: `status/state.json`, `previews/registry.json`.

### Approach C: Sync matching preview timestamps during `cycle_record.py --state-mode completed`
- Effort: M
- Risk: Low
- Pros: Removes the bookkeeping gap at the moment a cycle is finalized, keeps behavior tied to the state update flow already used by the lab.
- Cons: Slightly broader helper responsibility, needs careful tests so started-mode stays untouched.
- Reuses: `scripts/cycle_record.py`, `tests/test_cycle_record.py`, existing state and registry files.

## Recommended Approach
Choose Approach C. It is the smallest change that turns completed-cycle recording into a self-healing consistency step without introducing another command.
