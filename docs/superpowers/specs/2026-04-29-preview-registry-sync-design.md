# Design: Preview Registry Sync

Generated by brainstorming on 2026-04-29
Status: APPROVED (autonomous cron cycle)
Mode: Builder

## Problem Statement
The evolution lab keeps detecting drift between `status/state.json` published project timestamps and `previews/registry.json` preview metadata. That drift creates recurring health-cycle noise and leaves preview metadata stale even when the published artifact is current.

## What Makes This Cool
The smallest useful fix is self-healing metadata. A publish-related workflow should update the registry timestamp from one trusted source so the preview catalog stays truthful without manual bookkeeping.

## Premises
1. `status/state.json` is the current source of truth for published project freshness. Agree.
2. `previews/registry.json` should mirror `publishedProjects[].updatedAt` for matching preview paths when a matching preview exists. Agree.
3. The safest place to enforce this is the audit/helper layer already responsible for cycle consistency checks. Agree.

## Approaches Considered

### Approach A: Manual registry edit only
- Effort: S
- Risk: Medium
- Pros: Fastest immediate unblock, no code changes.
- Cons: Drift will recur, no guardrail, no repeatable fix.
- Reuses: Existing files only.

### Approach B: Audit detects and reports timestamp drift
- Effort: S
- Risk: Low
- Pros: Turns the silent mismatch into an explicit failing check with regression coverage.
- Cons: Still requires a human or later automation to repair drift.
- Reuses: `scripts/cycle_audit.py`, `tests/test_cycle_audit.py`.

### Approach C: Audit detects drift and offers/uses a sync helper
- Effort: M
- Risk: Low
- Pros: Both detects and repairs the mismatch, best long-term direction.
- Cons: Slightly more scope for one cycle because it touches behavior, tests, and cycle bookkeeping.
- Reuses: `scripts/cycle_audit.py`, preview registry structure, status metadata.

## Recommended Approach
Choose Approach C, but keep the implementation minimal: add explicit drift detection in `cycle_audit.py`, cover it with tests, and repair the current registry entry in this cycle record so the workspace ends healthy.

## Open Questions
- Whether future publish helpers should automatically rewrite the registry instead of only flagging mismatches.

## Next Steps

2. Implement the smallest audit logic needed to compare matching preview entries against `publishedProjects` timestamps.
3. Update the current registry entry to the published timestamp and verify the audit passes on the real workspace.
4. Record the cycle and update state metadata.
