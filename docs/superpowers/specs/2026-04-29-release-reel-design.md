# Design: Release Reel

## Problem Statement
Nik wants an aggressive cycle that lands a fresh public artifact, not more invisible maintenance. A phone-friendly static app that turns messy release notes into swipeable launch cards fits that brief: it is obvious in seconds, useful to makers, and safe to ship in one cycle without keys or backend risk.

## What Makes This Cool
Paste blunt changelog bullets, tap one button, and instantly get a cleaner launch deck inside a phone frame plus a copy-ready caption. The before and after contrast is easy to show from a phone and feels a little magical despite running fully client-side.

## Premises
1. The right problem is release presentation, not full marketing automation.
2. Doing nothing likely means missing the window for a new public repo today.
3. Existing lab constraints strongly favor a static browser app with zero backend and zero secrets.
4. The first release should focus on deterministic structure: parse bullets, group them into a clear narrative, and wrap them in a polished mobile-first UI.

## Approaches Considered

### Approach A: Plain changelog formatter
- Effort: S
- Risk: Low
- Pros: Fastest possible ship, easy logic.
- Cons: Weak wow factor, not visually impressive enough for the cycle goal.
- Reuses: existing static preview and GitHub Pages workflow.

### Approach B: Swipeable launch card studio (recommended)
- Effort: M
- Risk: Low
- Pros: Strong phone demo, visually clear value, still dependency-free, easy to test with pure modules.
- Cons: Slightly more UI work than a plain formatter.
- Reuses: existing static project pattern, Node built-in tests, preview mirror flow.

### Approach C: Full social asset exporter
- Effort: L
- Risk: Medium
- Pros: Highest ceiling for real-world reuse.
- Cons: Export/render/download polish would soak up the cycle and threaten the public ship.
- Reuses: only some of the same parsing logic.

## Recommended Approach
Choose Approach B. It keeps the scope tight while making the artifact obviously showable and more memorable than another utility page.

## Open Questions
- Should the first version support arbitrary numbers of cards? No. Keep it fixed to four cards for clarity and strong layout control.
- Should it try to rewrite every bullet heavily? No. Keep the rewrite light and deterministic so behavior stays trustworthy.

## Success Criteria
1. A stranger can understand the tool in under 30 seconds from the live demo.
2. The app works entirely client-side and reads well on a phone-sized screen.
3. Core parsing and deck generation have Node tests.
4. The project ships as a new public GitHub repo with GitHub Pages enabled.
5. The evolution preview registry includes a mirrored local preview.

## Next Steps
1. Build pure modules for parsing release notes and generating cards and caption copy, using explicit test-first steps.
2. Create a polished static UI with theme switching, example inputs, live deck rendering, and clipboard actions.
3. Publish the repo as `release-reel`, mirror it into the preview registry, and record the cycle with links.
