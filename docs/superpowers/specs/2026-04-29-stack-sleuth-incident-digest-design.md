# Design: Stack Sleuth Incident Digest

Generated by brainstorming on 2026-04-29
Status: APPROVED
Mode: Builder

## Problem Statement
Stack Sleuth already helps with one trace at a time, but real incidents often arrive as a pasted dump of repeated failures from logs, chat threads, or ticket comments. Developers need a fast way to collapse many traces into distinct incident groups, identify the dominant culprit signatures, and see which failures are repeats versus one-offs. That turns Stack Sleuth from a single-trace explainer into a lightweight debugging digest engine.

## What Makes This Cool
A user can paste a messy bundle of traces and instantly get an incident digest: how many traces were found, how many unique signatures they collapse into, the dominant runtime and error families, and the top repeated culprits with example summaries. It feels like a tiny log-triage analyst, not just a formatter.

## Premises
1. The best next step for Stack Sleuth is to go deeper on debugging workflow value, not start another unrelated static toy.
2. Repeated stack traces are a common real-world pain point, and signature grouping is a deterministic way to create obvious value without pretending to fully debug incidents.
3. The existing shared engine plus CLI architecture is the right base. New digest behavior should reuse the current single-trace parser, analyzer, and renderer instead of forking logic.
4. The browser experience can stay simple if the main value comes from a reusable digest engine and a stronger CLI/report surface.
5. The feature should gracefully handle mixed runtimes, duplicate traces, empty chunks, and noisy separators.

## Approaches Considered

### Approach A: Signature-grouped digest on top of the existing engine (recommended)
- Effort: M
- Risk: Low
- Pros: Reuses current analysis pipeline, creates a genuinely thicker engine, supports both CLI and browser, easy to explain in a demo.
- Cons: Digest quality depends on current signature heuristics, grouping stays deterministic rather than semantic.
- Reuses: `parseTrace`, `analyzeTrace`, current signature generation, current render patterns.

### Approach B: Session timeline view with per-trace cards
- Effort: M
- Risk: Medium
- Pros: More visual, easier to inspect each trace individually, could evolve into a fuller debugging workbench.
- Cons: UI-heavy, weaker CLI payoff, risks spending the cycle on presentation instead of reusable core logic.
- Reuses: current single-trace report UI and example loading patterns.

### Approach C: Cross-trace root-cause inference with heuristic merging
- Effort: L
- Risk: High
- Pros: Highest upside if it works, more magical summaries, more differentiated than plain grouping.
- Cons: Hard to validate in one cycle, higher false-confidence risk, more speculative than the current engine supports.
- Reuses: base parser and diagnosis tags, but needs substantially new logic.

## Recommended Approach
Choose Approach A. It compounds the strongest part of Stack Sleuth, creates a reusable engine surface, improves the CLI and browser together, and still has a clear same-cycle path to a public artifact.

## Design

### User-facing behavior
- Stack Sleuth should continue to support single-trace analysis exactly as it does now.
- When the input contains multiple traces, the app and CLI should produce an incident digest instead of forcing the user to reason about raw repeats.
- The digest should report total traces, unique incident groups, and ranked groups with repeat counts.
- Each group should surface the signature, runtime, error headline, likely culprit frame, confidence, tags, and a short summary.
- Each group should include one representative trace report so the user can still drill into a concrete example.

### Input model
- Introduce a trace-splitting layer that turns one pasted blob into ordered trace chunks.
- Support obvious separators such as blank-line blocks between stack traces and repeated runtime-specific headers.
- Ignore empty chunks and whitespace-only separators.
- Preserve raw chunk text for representative examples and future exports.

### Engine architecture
- Add a digest module that orchestrates chunk splitting, per-trace analysis, and grouping.
- Group analyzed traces by the existing `signature` field.
- Store per-group aggregates: count, first-seen index, representative report, runtimes, error names, confidence distribution, and merged tags.
- Sort groups by descending count, then by first appearance for stable output.
- Expose a reusable API that returns both the list of individual analyzed traces and the grouped digest summary.

### CLI behavior
- Add a `--digest` mode that always emits grouped incident output.
- Default text mode may auto-upgrade to digest output when multiple traces are detected, while single-trace behavior remains unchanged.
- Add JSON support for digest output so the tool can be scripted.
- Add Markdown support for digest output so users can paste concise incident summaries into tickets or chats.

### Browser behavior
- Update the UI copy to make multi-trace paste obvious.
- Show a digest summary panel when multiple traces are detected: total traces, unique incidents, and top repeated incidents.
- Preserve the existing single-trace card layout for representative per-group details instead of building an entirely new visual system.
- Add a bundled multi-trace example to demonstrate grouping value instantly.

### Error handling
- If no valid trace chunks are found, keep the current empty-state messaging.
- If only one valid trace is found, use the existing single-trace rendering path.
- If some chunks fail to parse cleanly, include them in the total only if they yield a non-empty analysis; otherwise drop them silently for now to keep the UX clean.

### Testing
- Add test-first coverage for trace chunk splitting across mixed runtimes and separators.
- Add digest engine tests for grouping, ordering, merged tags, count aggregation, and representative selection.
- Extend CLI tests for `--digest`, JSON digest output, and Markdown digest output.
- Extend README contract tests so new usage examples stay honest.

## Open Questions
- Should digest mode infer separators from timestamps or log prefixes? Not in this cycle. Stick to stack-trace-shaped chunk detection.
- Should digest groups merge near-miss signatures from line-number drift? Not yet. Exact signature grouping is more reliable for v1.
- Should the browser expose per-trace raw chunks? Not in the first pass. Representative reports are enough.

## Success Criteria
1. A user can paste several repeated traces and immediately see how many distinct incidents they represent.
2. The shared engine exposes reusable digest data, not just UI-specific formatting.
3. The CLI can emit digest output in text, JSON, and Markdown.
4. The browser demo clearly shows the difference between one trace and many traces.
5. Tests cover the new splitting and grouping logic with mixed-runtime examples.

## Next Steps

2. Implement a reusable digest engine on top of the existing analysis pipeline.
3. Wire digest rendering into the CLI and browser without regressing single-trace behavior.
